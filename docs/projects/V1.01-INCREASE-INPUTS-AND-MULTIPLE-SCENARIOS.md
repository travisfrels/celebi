# V1.01 Increase Inputs and Multiple Scenarios

| Status | Set On |
|--------|--------|
| Planning | 2026-03-14 |
| Active | 2026-03-14 |

## Context

### Situation

Celebi is a Tkinter desktop app (`src/app.py`, `src/probability_engine.py`) that calculates exact dice probabilities for Trench Crusade. Current constraints:
- **Pool size**: 1–6 (validated in `_validate_inputs`, spinbox range 2–6 in UI)
- **Pick count**: 1–3 (validated in engine, but UI only exposes 2 and 3 via radio buttons)
- **Algorithm**: Brute-force enumeration via `itertools.product(range(1,7), repeat=pool_size)` — O(6^N). At N=6, this is 46,656 outcomes (fast). At N=12, this is ~2.2 billion (infeasible).
- **UI**: Single scenario with one config panel (left) and one results table (right) in a 600x400 window
- **No success/failure summary**: The table shows per-total exact and cumulative probabilities but no aggregate threshold display

### Opportunity

The user needs larger dice pools (up to 12 dice), wider pick counts (1–4), a quick-read success/failure indicator with configurable threshold (default 7), and side-by-side scenario comparison (up to 4 scenarios). The current algorithm cannot scale to pool size 12, and the UI only supports a single configuration.

### Approach

Three changes are required:

1. **Replace the probability engine algorithm** with multiset enumeration using `itertools.combinations_with_replacement`. Instead of enumerating all 6^N permutations, enumerate unique sorted outcomes (multisets) and weight each by its multinomial coefficient. At N=12, this reduces 2.2B outcomes to 6,188 unique multisets — a ~350,000x reduction. Results remain exact.

2. **Add success/failure probability calculation** — a function that aggregates exact probabilities below and at/above a configurable threshold (default 7).

3. **Refactor the UI for multiple scenarios** — extract config+results into a reusable scenario panel, support 1–4 panels side by side with add/remove controls, display success/failure summary per scenario, change pick count to spinbox (1–4), expand pool size to 1–12.

#### Alternatives not chosen

- **Dynamic Programming** — Equal performance but higher implementation complexity and correctness risk for the top/bottom-K selection problem. The multiset space at N=12 is only 6,188, so DP's advantage of avoiding enumeration entirely provides no practical benefit.
- **Brute-force with higher cap** — Capping at pool_size=8 doesn't meet the N=12 requirement. Even N=10 (~60M outcomes) introduces noticeable latency.
- **Fixed success threshold (7)** — Always showing failure as 1–6 and success as 7+. A configurable threshold adds negligible implementation cost while supporting house rules and variants.
- **Fixed 2 scenarios** — Covers the user's stated example but limits more complex tactical analysis. Up to 4 with add/remove is more flexible without risking UI clutter.
- **Summary row in table** — Mixes aggregate and per-value data; a separate display above the table is more prominent and scannable.

#### Decisions

| Decision | Choice | Rationale |
|----------|--------|-----------|
| Algorithm | Multiset enumeration | O(C(N+5,5)) — 6,188 multisets at N=12 vs 2.2B brute-force outcomes. See [ADR-OP-PROBABILITY-CALCULATION-MULTISET-20260314](../adrs/ADR-OP-PROBABILITY-CALCULATION-MULTISET-20260314.md). |
| Pick count UI | Spinbox (1–4) | Radio buttons don't scale for 4 options; spinbox is consistent with pool size control |
| Success threshold | Configurable spinbox (default 7) | Trivial implementation cost, supports house rules |
| Multi-scenario layout | Side-by-side columns (up to 4) | Enables visual comparison; add/remove buttons for flexibility |
| Success/failure display | Separate element above table | More prominent and scannable than a summary row in the table |

## Goals

- Support pool sizes 1–12 and pick counts 1–4
- Display success/failure probability summary per scenario (configurable threshold, default 7)
- Allow up to 4 simultaneous scenarios for side-by-side comparison
- Replace the brute-force algorithm with multiset enumeration to handle larger pools
- Maintain exact combinatorial calculation (no simulation)

## Non-Goals

- Configurable dice types (still d6 only)
- Persistent storage / export of results
- Graphical charts or visualizations beyond the table
- Network/multiplayer features
- Full game rules engine (wounds, saves, morale, etc.)

## Exit Criteria

- [ ] Pool size spinbox allows 1–12; pick count spinbox allows 1–4 (constrained to not exceed pool size)
- [ ] Probability engine handles pool_size=12, pick_count=4 with correct exact probabilities in under 1 second
- [ ] Each scenario displays a success/failure summary (configurable threshold, default 7) showing aggregated probability below and at/above the threshold
- [ ] User can add up to 4 scenarios and remove any scenario (minimum 1 remains)
- [ ] Multiple scenarios display side by side with independent configurations
- [ ] All existing tests continue to pass
- [ ] New tests cover: pool sizes 7–12, pick count 4, success/failure calculation, multi-scenario UI interactions
