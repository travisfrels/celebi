# V1.02 Dark Mode and Dice Roller

| Status | Set On |
|--------|--------|
| Planning | 2026-03-22 |
| Active | 2026-03-22 |

## Context

### Situation

Celebi (V1.01) is a zero-dependency Python/Tkinter desktop app with:
- **Computation**: `src/probability_engine.py` — stateless multiset enumeration, returns `dict[int, float]` exact/cumulative probabilities
- **Presentation**: `src/app.py` — `CelebiApp` orchestrator + 1-4 `ScenarioFrame` instances (ttk widgets, grid layout, 1200x600 window)
- **Settings**: `.claude/settings.local.json` was copied from skiploom without adaptation — contains gradle, docker, playwright permissions irrelevant to a Python project
- **Styling**: No custom theming; relies entirely on ttk default system theme
- **No dice rolling**: The app is purely a probability calculator with no random number generation

### Opportunity

1. **Settings**: `.claude/settings.local.json` contains irrelevant permissions from skiploom (gradle, docker, playwright, npm, vitest). Should be tailored to celebi's Python/Tkinter stack.
2. **Dark mode**: No theme support. Windows 11 users with dark mode enabled see a jarring light-only app.
3. **Dice roller**: Players want to roll dice using their configured scenarios during gameplay, not just view probabilities. The probability calculator tells you *what could happen*; the roller tells you *what did happen*.

### Approach

Three changes are required:

1. **Settings cleanup** — Replace celebi's `.claude/settings.local.json` with celebi-appropriate permissions modeled after skiploom's structure: keep common entries (git, gh, WebSearch, WebFetch for relevant domains) and replace project-specific entries (gradle -> pytest, docker/playwright -> python).

2. **Dark mode via ttk.Style + winreg** — Detect the Windows system theme at startup using `winreg` (stdlib) and apply matching light/dark color palettes via `ttk.Style`. Maintains the zero-dependency constraint from ADR-OP-GUI-FRAMEWORK-20260313.

3. **Dice roller embedded in ScenarioFrame** — Add a Roll button, visual d6 face display (tk.Canvas), and sum label to each ScenarioFrame. Rolling logic lives in a new `src/dice_roller.py` module, separate from the probability engine.

#### Alternatives not chosen

- **sv_ttk** — Polished Win11 Sun Valley theme with built-in dark mode, but breaks the zero-dependency constraint.
- **ttkthemes** — Generic dark themes, not OS-aware, adds dependency for marginal benefit.
- **darkdetect + manual styling** — Adds dependency for OS detection only; same manual styling work as the chosen approach.
- **Shared dice roller panel below scenarios** — Scenario selector adds indirection and friction. Breaks ScenarioFrame self-containment.
- **Separate dice roller window** — Popups feel disconnected from scenarios. Three-step workflow during gameplay is too slow.

#### Decisions

| Decision | Choice | Rationale |
|----------|--------|-----------|
| Theme detection | `winreg` (stdlib) | Reads `AppsUseLightTheme` from Windows registry. Zero dependencies. |
| Theme application | `ttk.Style` | Configure widget colors for light/dark palettes. Covers all ttk widgets used. |
| Theme timing | Startup detection only | Read registry once at app init. No runtime listener for V1.02. |
| Dice roller placement | Within each ScenarioFrame | One-click rolling, zero ambiguity about which config drives the roll. Fits self-contained architecture. |
| Dice rendering | tk.Canvas with dot patterns | Draw d6 faces as small squares with circle dots. Stdlib only. |
| Rolling logic | New `src/dice_roller.py` module | Separate from probability engine (random vs combinatorial). Testable independently. |

## Goals

- Align `.claude/settings.local.json` with celebi's actual tech stack
- Detect Windows light/dark mode and apply matching theme at startup
- Enable dice rolling within each scenario using that scenario's configuration
- Display individual die faces and the sum of selected dice after rolling

## Non-Goals

- Custom theme editor or persistent theme preference
- Runtime theme change detection (listening for OS theme switches)
- Dice animation or sound effects
- Non-d6 dice types
- Cross-platform dark mode detection (Linux, macOS)

## Exit Criteria

- [ ] `.claude/settings.local.json` contains only celebi-relevant permissions (python, pytest, git, gh) with no skiploom-specific entries (gradle, docker, playwright, npm)
- [ ] App detects Windows light/dark mode at startup and applies matching theme to all widgets
- [ ] Light theme and dark theme are visually distinct with readable contrast for all widget types (labels, spinboxes, buttons, treeview, frames)
- [ ] Each scenario has a "Roll" button that generates random dice using that scenario's pool size, selection strategy, pick count, and modifier
- [ ] After rolling, each die face is displayed as a visual d6 (dot pattern) within the scenario panel
- [ ] After rolling, the sum of selected dice (with modifier applied) is displayed
- [ ] Selected dice are visually distinguished from unselected dice in the pool
- [ ] All existing tests (93) continue to pass
- [ ] New tests cover: theme detection/application, dice roller logic, dice roller UI integration
